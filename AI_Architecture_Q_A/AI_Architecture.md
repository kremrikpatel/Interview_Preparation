These test senior architectural judgment, deep system design, and the ability to reason under ambiguity.

Agentic AI & Orchestration
--------------------------

**Q1. How do you design a multi-agent system where agents can disagree and reach a consensus without infinite loops?**

**A:** Use a _critic-arbiter pattern_: specialized agents produce outputs, a critic agent scores them against defined rubrics, and a neutral arbiter agent applies a deterministic rule (majority vote, confidence threshold, or escalation to human-in-the-loop) to break deadlock. Prevent infinite loops by imposing a max-iteration budget per task, tracking agent state in a shared context store (e.g., Redis or a vector DB), and implementing circuit-breaker logic that escalates to human review after N failed rounds.​

**Q2. Explain how you would architect an agentic system that safely calls tools with side effects (e.g., payments, deletions) in a production enterprise environment.**

**A:** Implement a _tool permission layer_ with scoped, least-privilege credentials per agent role. Use a pre-execution approval flow for destructive/financial operations — the agent proposes an action, a human or governance agent confirms, and only then does execution proceed. All tool calls are logged to an immutable audit trail. Rollback handlers must be registered for every side-effectful tool. Idempotency keys prevent duplicate execution on retries.​

**Q3. How do you prevent prompt injection in a multi-tenant AI SaaS platform where user data is embedded into prompts?**

**A:** Apply a layered defence: (1) structural separation — never concatenate raw user input into system prompts; use template slots with strict schema validation; (2) input sanitization — strip HTML/markdown and detect adversarial patterns using a fine-tuned classifier or rule engine; (3) output validation — scan LLM outputs for instruction-following anomalies before returning to the user; (4) privilege separation — route different trust levels through different model endpoints with separate system contexts. Monitor with prompt-layer tracing tools like LangSmith or Weights & Biases.

**Q4. A RAG system returns factually inconsistent answers for the same query on different calls. How do you diagnose and fix this?**

**A:** Non-determinism sources: (1) top-k retrieval fetching different chunks due to approximate nearest-neighbor (ANN) index variance — fix with deterministic retrieval (exact search, or seed-locked ANN); (2) LLM temperature > 0 — set temperature=0 for factual tasks; (3) chunk boundary inconsistency — retrieved context differs because documents were chunked without overlap. Implement an _answer consistency evaluator_ that runs the same query N times and flags semantic divergence above a threshold. Add citation grounding to make retrieved source traceable per answer sentence.

**Q5. How would you architect a zero-downtime migration of a production LLM pipeline from GPT-4 to an open-source self-hosted model?**

**A:** Use a _shadow deployment_ pattern: run both models in parallel behind a traffic router. Initially send 100% to GPT-4 and 0% to the new model, but log both outputs. Run automated evaluation (BLEU, embedding similarity, task-specific metrics) on shadow outputs for 2–4 weeks. Gradually shift traffic using feature flags (1% → 10% → 50% → 100%) while monitoring latency, cost, and quality regressions. Maintain a fast rollback path via the feature flag. Never do a hard cutover without a 30-day parallel-run baseline.

**Q6. Design an AI observability architecture for a system with 50+ LLM-powered microservices.**

**A:** Instrument each service with structured trace spans using OpenTelemetry, capturing: prompt hash, token counts, model version, latency, retrieved chunk IDs, and confidence scores. Funnel traces to a central collector (e.g., Grafana Tempo or Datadog). Build model-specific dashboards tracking: response latency p50/p95/p99, cost-per-request, hallucination rate (via automated factuality scorer), and embedding drift (cosine similarity decay over time). Alert on anomaly thresholds per service SLA. Use LangSmith or Langfuse for LLM-specific tracing with prompt versioning.

**Q7. How do you design an AI data architecture that handles both real-time inference and batch model training on the same dataset?**

**A:** Use a _Lambda architecture variant_ with a unified serving layer: a streaming layer (Kafka/Pub-Sub → feature store like Feast) for real-time feature serving, and a batch layer (Spark/BigQuery → data lake) for training. Both layers write to a _feature store_ that provides consistent feature definitions. The model training pipeline reads from the batch layer; inference reads from the real-time layer using the same feature logic, avoiding training-serving skew. Version features explicitly — skew between training and serving features is the #1 silent failure mode in production ML.

**Q8. Explain how you'd implement semantic caching for LLM responses and the trade-offs involved.**

**A:** Semantic caching stores previous query-response pairs indexed by embedding. On a new query, compute its embedding and search for cache hits above a cosine similarity threshold (e.g., 0.95). If found, return the cached response, saving inference cost and latency. Trade-offs: (1) _staleness_ — cached answers may become outdated as the underlying knowledge base changes; mitigate with TTL expiry tied to data freshness SLAs; (2) _threshold calibration_ — too low a threshold returns wrong answers for superficially similar but semantically different queries; (3) _multi-tenant safety_ — cache partitioning required to prevent cross-tenant data leakage.

**Q9. How do you design an evaluation framework for a non-deterministic generative AI system in a regulated industry (e.g., finance or healthcare)?**

**A:** Build an _automated + human-in-loop_ evaluation pipeline: (1) _automated metrics_ — factuality score (using an LLM-as-judge approach against a golden reference), coherence, toxicity, and domain-specific compliance checks (e.g., no PII in outputs, no unqualified financial advice); (2) _red-teaming_ — adversarial prompt battery testing run on every model update; (3) _human review sampling_ — 2–5% of production outputs reviewed by domain SMEs, with feedback loops into fine-tuning; (4) _regression test suite_ — a curated set of 500+ benchmark Q&A pairs where any score drop > 2% triggers rollback.

**Q10. Describe how you would architect a private, enterprise-grade RAG system where documents must never leave the corporate network.**

**A:** Deploy a fully on-premises or private VPC stack: (1) document ingestion pipeline runs in a private subnet with no egress; (2) embedding model runs locally (e.g., sentence-transformers on CPU, or a fine-tuned model on private GPU); (3) vector store is self-hosted (pgvector on Cloud SQL Private, Weaviate on GKE, or Qdrant); (4) LLM inference runs on-prem (Ollama, vLLM, or Azure Private Endpoints for Azure OpenAI with no data logging enabled). Enforce network policies via Kubernetes NetworkPolicy or VPC Service Controls on GCP to prevent exfiltration.

**Q11. How do you handle LLM context window limitations in a system that must reason over 500-page enterprise documents?**

**A:** Don't try to fit everything in context. Use _hierarchical RAG_: (1) chunk documents into paragraphs with 10–20% overlap; (2) create a secondary index of chunk summaries for coarse retrieval; (3) on query, first retrieve top-N summary chunks, then retrieve detailed chunks from within those sections; (4) use _re-ranking_ (cross-encoder or Cohere Rerank) to select the most relevant 3–5 chunks for the final prompt. For truly sequential reasoning (e.g., contract comparison), use _map-reduce prompting_: process each section independently, then synthesize a final answer from section-level outputs.

**Q12. A Kubernetes-hosted LLM inference service is experiencing GPU memory fragmentation causing OOM crashes. How do you resolve it?**

**A:** Root causes: (1) variable-length batching without padding constraints; (2) KV cache growing unbounded. Solutions: (1) use _PagedAttention_ (vLLM's core feature) to manage KV cache in non-contiguous memory pages, eliminating fragmentation; (2) set max\_model\_len and max\_num\_seqs to cap concurrent request memory; (3) enable dynamic batching with a token budget rather than a request count budget; (4) configure GPU memory utilization limits (e.g., gpu\_memory\_utilization=0.85 in vLLM); (5) add Kubernetes LimitRange on GPU memory requests and use node auto-provisioning with dedicated GPU node pools.

**Q13. How do you design a cost-aware AI inference architecture that dynamically routes requests to cheaper models without degrading quality?**

**A:** Implement an _LLM router_ (inspired by tools like RouteLLM or LiteLLM): classify each incoming query by complexity using a lightweight classifier (e.g., a fine-tuned BERT or a rule-based heuristic on query length/domain). Route simple queries (FAQ, summarization) to a cheap fast model (GPT-4o-mini, Haiku), and complex queries (multi-hop reasoning, code generation) to a powerful model (GPT-4o, Sonnet 3.7). Log model selection decisions and actual quality outcomes, then retrain the router periodically. Target 60–80% of traffic to cheap models to achieve 40–60% cost reduction.

**Q14. Explain the CAP theorem's implications specifically for a distributed AI vector search system under high write throughput.**

**A:** In a distributed vector store (e.g., Weaviate cluster, Pinecone), under network partition you must choose between Consistency (all nodes return the same index state) and Availability (all nodes respond, even if stale). For AI search, _AP (Availability + Partition tolerance)_ is typically the right trade-off: a slightly stale index is far better than query timeouts. However, for high write throughput during document ingestion, use _eventual consistency with write buffering_ — batch writes go into a Kafka topic, an async indexing worker processes them, and the search index catches up within seconds. Monitor index lag as a key SLA metric.

**Q15. How do you implement responsible AI guardrails at the architecture level — not just the prompt level — for a customer-facing chatbot?**

**A:** Guardrails must be defence-in-depth: (1) _input layer_ — PII detector strips/masks sensitive data before it hits the LLM; topic classifier blocks out-of-scope queries; (2) _model layer_ — system prompt constraints + model-level safety settings (OpenAI moderation API, Azure content filters, Guardrails AI); (3) _output layer_ — a post-generation validator checks for PII leakage, toxicity, legal liability phrases, hallucinated URLs; (4) _logging layer_ — every interaction is logged with user ID for audit; (5) _human review layer_ — flagged outputs are queued for human review with RLHF feedback loop.

System Design & Distributed Architecture
----------------------------------------

**Q16. How would you design an event-driven AI pipeline that processes 1M document ingestion events per day with guaranteed delivery?**

**A:** Use a durable message queue (Kafka or Pub/Sub) as the backbone. Document upload events publish to a raw-documents topic. A consumer group of Cloud Run workers pulls events, chunks and embeds documents, then publishes to an embeddings-ready topic consumed by the vector DB writer. Use _at-least-once delivery_ with idempotency keys (document hash) to prevent duplicate indexing. Dead-letter queues (DLQ) capture failed events for reprocessing. Monitor consumer lag as the primary SLA metric — if lag exceeds 10 minutes, auto-scale the worker pool.

**Q17. Design a multi-tenant AI platform where tenant A's data must be cryptographically isolated from tenant B's, even with shared infrastructure.**

**A:** Use _envelope encryption with per-tenant KEKs (Key Encryption Keys)_: each tenant has a unique KEK stored in a KMS (AWS KMS, GCP Cloud KMS). Data stored in shared storage is encrypted with a Data Encryption Key (DEK) that is itself encrypted with the tenant's KEK. The application decrypts the DEK at query time using the tenant's KEK — without the KEK, the data is unreadable even if the storage layer is shared. For vector stores, use namespace-scoped indices per tenant (Pinecone namespaces, Weaviate multi-tenancy, pgvector with Row Level Security). This provides logical and cryptographic isolation without requiring physical infrastructure separation.

**Q18. How would you architect an AI solution that needs sub-100ms inference latency at the 99th percentile for a real-time customer interaction?**

**A:** Sub-100ms at p99 for LLM inference is extremely challenging — full generation rarely achieves this. Strategies: (1) _speculative decoding_ — a small draft model generates tokens, the large model verifies in parallel, achieving 2–3x speedup; (2) _streaming with first-token latency_ — target < 200ms time-to-first-token (TTFT), which gives the user perceived responsiveness; (3) _model distillation_ — use a fine-tuned small model (7B) for the specific domain, achieving comparable quality with 5–10x lower latency; (4) _semantic cache hit_ — cache common queries to bypass inference entirely; (5) co-locate the model on GPU instances in the same cloud region as the application. For truly < 100ms, avoid generative LLMs and use embedding + classification models instead.

**Q19. What is your approach to API versioning for AI services where model behaviour itself changes with model upgrades?**

**A:** AI APIs need _behavioural versioning_, not just schema versioning. Embed the model version in the API contract: POST /v2/completions with header X-Model-Version: gpt-4o-2025-03. Maintain version pinning — consumers explicitly opt into model upgrades rather than receiving silent behavioural changes. When upgrading, run _A/B evaluation_ before promoting. Maintain at minimum 3 previous model versions in production with a 90-day deprecation window. Document behavioural change logs alongside schema change logs — this is a capability gap in most OpenAPI specs and must be addressed explicitly in AI service governance.

**Q20. How do you design a data pipeline that avoids training-serving skew in a feature store used for both ML training and real-time inference?**

**A:** Training-serving skew occurs when feature computation logic differs between the batch training pipeline and the real-time serving path. Eliminate it by: (1) defining _feature transformation logic once_ in the feature store (e.g., Feast, Tecton) — both training and serving call the same feature retrieval function; (2) using _point-in-time correct joins_ during training to ensure features reflect what was known at the time of the label, not future data; (3) logging live inference feature values to a feature store, then periodically running statistical tests (PSI, KL divergence) to detect distribution shift between logged live features and the training distribution.

**Q21. Describe a threat model for an LLM-powered internal enterprise assistant with access to HR and financial data.**

**A:** Key threats: (1) _prompt injection via user input_ — attacker embeds instructions to exfiltrate data; mitigate with input sanitization and output monitoring; (2) _indirect prompt injection via documents_ — malicious instructions embedded in HR documents retrieved by RAG; mitigate by sandboxing retrieval context and tagging it as untrusted; (3) _privilege escalation_ — user queries for data beyond their role; mitigate with pre-query RBAC check before retrieval; (4) _data leakage via model memorization_ — LLM trained or fine-tuned on HR data might reproduce PII; use differential privacy training techniques; (5) _insider threat via admin access to logs_ — encrypt audit logs and restrict decryption to security team.

**Q22. How do you architect an AI system for compliance with Australian Privacy Act 2024 and GDPR simultaneously?**

**A:** Implement _data residency controls_ — Australian user data processed on Australian infrastructure (GCP australia-southeast1), EU data on EU infrastructure, with no cross-border replication by default. For both regimes: (1) _data minimisation_ — collect only what's needed for the AI use case; (2) _right to erasure_ — implement a deletion propagation pipeline that removes user data from training sets, vector stores, and logs; (3) _purpose limitation_ — data collected for one AI feature cannot be reused for another without consent; (4) _explainability log_ — for consequential AI decisions (e.g., loan assessment), store the retrieval context and prompt that generated the decision for audit purposes.

**Q23. Explain how you would design a self-healing AI pipeline that automatically recovers from model degradation in production.**

**A:** Implement _automated quality monitoring with remediation triggers_: (1) a quality probe runs every 15 minutes, sending a synthetic golden test set to the model and scoring outputs; (2) if quality drops below a threshold (e.g., factuality score < 0.85), an automated runbook triggers: first, roll back to the previous model version; second, alert the AI ops team; third, begin a root cause analysis by comparing prompt hashes and context distributions; (4) if rollback doesn't restore quality (e.g., due to data drift, not a model change), trigger an automated retraining pipeline with the latest data. This forms a _closed-loop AI operations_ system.

**Q24. How do you architect vector search at scale — 100M+ vectors — with low query latency?**

**A:** At 100M+ vectors, exhaustive search is infeasible. Architecture: (1) use _Hierarchical Navigable Small Worlds (HNSW)_ index for sub-10ms approximate nearest neighbour search — supported by Weaviate, Qdrant, pgvector with HNSW extension; (2) shard vectors across multiple nodes by document collection or tenant — route queries to the correct shard with a metadata filter pre-step; (3) use _quantization_ (product quantization or scalar quantization) to compress vector dimensions 4–16x, reducing memory and improving cache hit rate; (4) warm the HNSW graph in memory on node startup — cold starts cause 10–100x latency spikes; (5) separate the _hot_ (recent/frequent) index from the _cold_ (archival) index, with tiered query routing.

**Q25. How do you design a GitOps-based AI model deployment pipeline with blue/green support and automated rollback?**

**A:** Store model configuration (version, serving parameters, resource quotas) in Git as Helm values or Kustomize overlays. ArgoCD or Flux syncs the Git state to the cluster. On a new model version: (1) deploy to the _green_ slot — a separate Kubernetes deployment with its own service endpoint; (2) automated evaluation job runs against the green endpoint; (3) if evaluation passes, a weighted traffic split (VirtualService in Istio or HTTPRoute) shifts traffic from blue to green incrementally; (4) if any alert fires (latency, quality, error rate), ArgoCD's automated sync reverts the Helm values to the previous version, restoring the blue slot instantly. Model weights themselves are stored in an object store (GCS/S3) versioned by digest hash.

**Q26. How do you optimise GPU utilisation for LLM inference on Kubernetes at enterprise scale?**

**A:** Key levers: (1) _continuous batching_ — use vLLM's continuous batching to process multiple requests on the same GPU pass, achieving 2–4x throughput improvement over naive request-per-GPU approaches; (2) _bin-packing pod scheduling_ — use Kubernetes topologySpreadConstraints + custom resource nvidia.com/gpu requests to pack multiple smaller model replicas onto the same node where VRAM allows; (3) _MIG (Multi-Instance GPU)_ — on A100/H100, partition a single GPU into 2–7 isolated instances, each running a separate model replica; (4) _spot/preemptible GPU nodes_ for batch inference workloads with checkpointing; (5) _scale-to-zero_ for development/staging GPU nodes using Karpenter or GKE Autopilot.

**Q27. You're asked to reduce AI infrastructure spend by 40% without degrading quality. Walk through your FinOps approach.**

**A:** Systematic cost reduction: (1) _LLM router_ — 60–70% of queries to cheaper models (saves 50–80% per token); (2) _semantic caching_ — 20–40% cache hit rate eliminates inference entirely for those queries; (3) _batched async processing_ — non-real-time workloads run on scheduled batch jobs using spot instances, 60–80% cheaper; (4) _token optimisation_ — audit system prompts for verbosity; reducing prompt length by 30% reduces cost by 30%; (5) _reserved instance planning_ — commit to 1-year GPU reservations for baseline load; (6) _embedding model self-hosting_ — replacing OpenAI text-embedding-3-large with a self-hosted bge-large on a single A10G GPU saves ~$800/month at 10M embeddings/day.

Leadership, Governance & Strategy
---------------------------------

**Q31. How do you make a build-vs-buy decision for a core AI component like a vector database?**

**A:** Evaluate on 5 axes: (1) _strategic differentiation_ — does this component differentiate the product? If not, buy/use managed; (2) _operational cost_ — self-hosting a vector DB requires 0.5–1 FTE of platform engineering; compare against managed service pricing at your scale; (3) _data sovereignty_ — regulatory requirements may mandate self-hosting; (4) _scalability ceiling_ — does the managed service meet your 3-year growth projection? Check vendor pricing cliffs; (5) _lock-in risk_ — assess migration cost if the vendor changes pricing or shuts down. For most enterprises, managed services (Pinecone, Weaviate Cloud) win at < 100M vectors; self-hosted wins at > 500M vectors or strict data residency requirements.

**Q32. How do you communicate a complex AI architecture decision to a non-technical C-suite audience?**

**A:** Use the _business outcome → trade-off → recommendation_ structure. Never lead with technology. Example: "We chose to self-host our AI models rather than use an external API. This costs $200K more upfront, but it means customer data never leaves our network — eliminating a critical compliance risk that would have blocked our enterprise sales pipeline. The trade-off was speed-to-market; we accept 3 months of additional delivery time in exchange for removing a deal-blocking risk." Use a one-page Architecture Decision Record (ADR) with three columns: Option, Cost/Risk, Recommendation. Attach a cost model in a spreadsheet — executives trust numbers.

**Q33. Walk me through how you'd write an Architecture Decision Record (ADR) for choosing between LangChain and Semantic Kernel for an enterprise agent platform.**

**A:** ADR structure: (1) _Context_ — we need an agent orchestration framework for a multi-agent enterprise platform with .NET and Python services; (2) _Decision Drivers_ — language polyglotism, Microsoft ecosystem integration, community support, production stability; (3) _Options Considered_ — LangChain (Python-first, broad ecosystem, rapid but unstable API), Semantic Kernel (Microsoft-backed, polyglot, enterprise-focused), LlamaIndex (RAG-optimised, less agent-native), custom framework; (4) _Decision_ — Semantic Kernel for .NET services, LangChain for Python data science workloads; (5) _Consequences_ — team upskilling required on Semantic Kernel; positive: better Azure OpenAI integration and Microsoft enterprise support SLA.

**Q34. A delivery team is consistently bypassing your architectural standards to ship faster. How do you handle this?**

**A:** Don't treat this as a compliance problem — treat it as a _friction problem_. First, audit which standards are being bypassed and why: if a team skips the security review because it takes 3 weeks, the standard is the problem, not the team. Introduce _paved roads_: pre-approved patterns, reference architectures, and golden path templates that make the right thing the easy thing. Move architecture reviews earlier (shift-left) into the sprint planning phase, not the delivery gate. Create an architectural working group with rotating engineering leads — when engineers co-own the standards, adoption follows. Reserve hard gates only for data privacy, security, and compliance — everything else should be guidance, not a mandate.

**Q35. How do you measure the success of an AI architecture you've designed?**

**A:** Success metrics span four dimensions: (1) _technical health_ — p95 latency vs. SLA, error rate, model quality score, infrastructure cost per query; (2) _delivery velocity_ — cycle time from AI feature idea to production, number of teams self-serving the reference architecture; (3) _business impact_ — revenue influenced, customer satisfaction delta, process automation hours saved; (4) _risk posture_ — number of security findings in AI audit, compliance certifications maintained, data incident count. Review these in a quarterly Architecture Health Review. A good architecture should make the next feature faster to build, not slower — if teams are getting slower over time, there is hidden technical debt in the architecture.

**Q51. What is RAG and how does it differ from fine-tuning?**

**A:** RAG (Retrieval-Augmented Generation) injects relevant documents from an external knowledge base into the LLM's prompt at query time — the model's weights never change. Fine-tuning updates the model's weights on a curated dataset, baking in new knowledge or behaviour permanently. RAG is preferred when knowledge changes frequently (documents updated daily), when you need citations, or when the knowledge base is too large for fine-tuning. Fine-tuning is preferred when you need to change the model's _style, tone, or output format_, or when you want domain-specific reasoning baked in rather than retrieved.

**Q52. Explain the difference between semantic search and keyword search, and when you'd use each.**

**A:** Keyword search (BM25, Elasticsearch) matches exact terms — fast, cheap, interpretable. Semantic search uses embedding vectors to match meaning, not exact words — finds "vehicle accident" for the query "car crash." Use keyword search for structured data, known terminology, and high precision requirements. Use semantic search for natural language queries, cross-lingual retrieval, and when users won't know exact terminology. _Hybrid search_ (combining BM25 + vector, re-ranked with a cross-encoder) outperforms both in most enterprise scenarios and is the current production standard.

**Q53. What is the difference between LangChain, LlamaIndex, and Semantic Kernel?**

**A:** LangChain is a general-purpose Python framework for building LLM chains and agents — broad ecosystem, rapid community development, but historically unstable APIs. LlamaIndex (formerly GPT Index) specialises in data ingestion, indexing, and RAG pipelines — it excels at connecting LLMs to external data sources with rich chunking and retrieval strategies. Semantic Kernel is Microsoft's enterprise-focused SDK supporting Python, C#, and Java — designed for integration with Azure OpenAI, Microsoft 365, and enterprise .NET systems. For a GCP-native Python shop, LangChain + LlamaIndex is the natural fit; for Azure + .NET, Semantic Kernel wins.

**Q54. What is a vector embedding and how is it generated?**

**A:** An embedding is a dense numerical vector (e.g., 1536 dimensions) that represents the semantic meaning of text. Generated by a neural encoder model (e.g., text-embedding-3-large, bge-large-en, sentence-transformers) that processes input text and outputs a fixed-length vector where semantically similar texts produce similar vectors (high cosine similarity). The quality of retrieval in a RAG system depends directly on embedding model quality — a fine-tuned domain-specific embedding model typically outperforms a general-purpose model for specialised corpora (e.g., legal, medical).

**Q55. What is the difference between synchronous and asynchronous AI inference, and when do you use each?**

**A:** Synchronous inference: the client waits for the model response — appropriate for real-time user interactions where latency < 5 seconds is expected. Asynchronous inference: the client submits a job and polls or receives a callback when complete — appropriate for batch document processing, background enrichment, or long-running generation tasks (> 30 seconds). For LLMs with streaming, use _server-sent events (SSE)_ to stream tokens synchronously as they're generated — this is the standard pattern for chat UIs, giving users perceived responsiveness even for 20-second full responses.

**Q57. What is a knowledge graph and when would you augment RAG with one?**

**A:** A knowledge graph (KG) represents entities and their relationships as nodes and edges. Standard RAG retrieves unstructured text passages; it struggles with multi-hop relational questions like "Who is the CEO of the company that acquired Firm X in 2023?" GraphRAG (Microsoft's approach) builds a KG from the document corpus and traverses entity relationships for complex queries. Use GraphRAG when: (1) queries require multi-hop reasoning across entities; (2) the corpus contains highly interconnected domain knowledge (e.g., financial networks, drug interactions); (3) precise entity disambiguation is critical. Standard RAG suffices for factual document Q&A without relational complexity.

**Q58. Explain the ReAct (Reasoning + Acting) pattern for AI agents.**

**A:** ReAct interleaves _Thought_ (the agent's reasoning about what to do next) and _Action_ (calling a tool) in an iterative loop until the task is complete. The cycle: Thought → Action → Observation → Thought → Action → Observation → ... → Final Answer. This makes the agent's reasoning transparent and debuggable. Compared to pure chain-of-thought (reasoning only) or pure action (tool calls without reasoning), ReAct produces more reliable outputs on multi-step tasks because the agent evaluates tool results before deciding the next action. It's the default pattern in LangChain agents and forms the basis of most production agent frameworks.​

**Q59. What is the difference between an orchestrator agent and a sub-agent?**

**A:** The orchestrator (or supervisor) agent manages the overall task: it decomposes goals, assigns subtasks to specialised sub-agents, tracks state, aggregates results, and decides when the task is complete. Sub-agents are specialised workers — a "web search agent," a "code execution agent," a "document summarisation agent" — each with a narrow scope and specific tool access. This separation of concerns improves maintainability and security: sub-agents have only the permissions they need (least-privilege). The orchestrator never executes tools directly — it delegates. This is the multi-agent pattern used in AutoGen, CrewAI, and LangGraph.​​

**Q60. How do you implement memory in an AI agent?**

**A:** Agent memory has four types: (1) _in-context (short-term)_ — the conversation history in the current prompt window; (2) _external episodic (long-term)_ — past interactions stored in a vector DB, retrieved by semantic similarity at the start of each session; (3) _semantic memory_ — a knowledge base the agent can query (RAG); (4) _procedural memory_ — stored prompt templates or skills the agent can reuse. For enterprise agents, combine all four: the context window holds the current task state; a Redis cache holds session state; a vector DB holds long-term episodic memory; a tool registry holds procedural knowledge.

**Q61. What is the difference between REST, GraphQL, and gRPC for AI service APIs — and which do you use when?**

**A:** Each protocol solves a different problem in AI service integration:​

**Q62. How do you design an API gateway pattern specifically for AI services?**

**A:** An AI API gateway handles concerns that standard gateways don't: (1) _token budget enforcement_ — reject requests where the estimated prompt + completion token count exceeds a per-tenant budget; (2) _model routing_ — inspect request metadata and route to the appropriate model endpoint (complexity-based routing); (3) _semantic rate limiting_ — not just request-per-second but tokens-per-minute per tenant (matching OpenAI's TPM model); (4) _prompt logging_ — log all requests/responses to an immutable audit store before forwarding to the model; (5) _streaming passthrough_ — proxy SSE token streams directly to the client without buffering, preserving perceived latency. Tools: Kong AI Gateway, AWS API Gateway + Lambda authorizer, or Apigee with custom policies on GCP.

**Q63. How do you version AI APIs when the underlying model behaviour changes?**

**A:** Standard semantic versioning (v1, v2) alone is insufficient for AI APIs because the _same schema can return different outputs_ after a model upgrade. Best practice: use _compound versioning_ — POST /v2/generate with a required model\_version header (e.g., gpt-4o-2025-03). Clients that pin a model version get deterministic behaviour. Clients that use latest accept behavioural evolution. Maintain a _breaking change policy_: model upgrades that degrade benchmark scores by > 5% on your golden test set are treated as breaking changes requiring a major version bump. Document behavioural changelogs alongside schema changelogs in your API portal.​

**Q64. How do you design structured output parsing from LLMs reliably in production?**

**A:** LLMs produce free-text by default — production systems require structured, machine-parseable output. Strategies in reliability order: (1) _JSON mode / response format enforcement_ — OpenAI and Anthropic support response\_format: json\_object or typed schemas, forcing valid JSON output; (2) _Pydantic output parsers_ — define your expected schema as a Pydantic model, inject the schema into the prompt as instructions, and parse the output with automatic retry on validation failure; (3) _constrained decoding_ — tools like Outlines or Guidance constrain the token sampling to only emit valid tokens for the target schema, guaranteeing valid output at the model level — the most reliable approach for critical pipelines; (4) _always wrap parsing in a retry loop_ with a max-3-attempt budget and a fallback to a simpler schema on failure.

Cloud Infrastructure & Kubernetes
---------------------------------

**Q65. When would you use Cloud Run versus GKE for LLM inference workloads?**

**A:** Cloud Run suits CPU/small GPU inference (e.g., embedding generation, small classifiers) where the workload is bursty, traffic is unpredictable, and you want scale-to-zero cost savings. It supports GPU instances (L4, T4) on Cloud Run GPU, but has a cold-start penalty of 10–30 seconds for large model loading — unacceptable for real-time inference SLAs. GKE is the right choice for: large GPU models (> 7B parameters) where keeping instances warm is cost-justified; multi-GPU serving (tensor parallelism requiring NVLink); fine-grained scheduling (GPU node pools, MIG partitioning); and stateful workloads needing persistent storage for model weights via PVCs. Rule of thumb: use Cloud Run for embeddings and small models; use GKE for production LLM inference above 13B parameters.

**Q66. How do you manage Persistent Volume Claims (PVCs) for LLM model weights in Kubernetes?**

**A:** Model weights (10–700GB per model) must not be loaded from object storage on every pod start — this causes 5–30 minute cold starts. Architecture: (1) provision a ReadOnlyMany PVC backed by a high-throughput storage class (GCP Filestore Premium, AWS EFS); (2) a one-time _model loader init container_ pulls weights from GCS/S3 into the PVC on first run; (3) all inference pods mount the PVC in ReadOnly mode — multiple replicas share the same pre-loaded weights; (4) use a PVC snapshot strategy for model version rollback — take a VolumeSnapshot before updating weights; (5) for models > 100GB, use _node-local SSDs_ (local-ssd storage class) for fastest I/O, with a daemonset that pre-caches weights on each GPU node.

**Q67. How does Kubernetes Horizontal Pod Autoscaler (HPA) need to be configured differently for LLM inference versus standard microservices?**

**A:** Standard HPA scales on CPU/memory — both are poor proxies for LLM inference load. A GPU pod can be CPU-idle but GPU-saturated. Correct approach: (1) use _custom metrics HPA_ with KEDA (Kubernetes Event-Driven Autoscaling) — scale on queue depth (Pub/Sub message backlog, Kafka consumer lag) for async inference; (2) for synchronous inference, scale on _active request count_ or _GPU utilisation_ via the DCGM Prometheus exporter metric DCGM\_FI\_DEV\_GPU\_UTIL; (3) set stabilizationWindowSeconds to 300 seconds on scale-down — GPU pods are expensive to cold-start, so avoid thrashing; (4) configure minReplicas: 1 in production (never scale-to-zero for synchronous inference) and maxReplicas based on your GPU node pool capacity.

**Q68. What is Terraform state management and why does it matter for AI platform infrastructure?**

**A:** Terraform state is the file that maps your HCL configuration to real cloud resources. For AI platforms with complex dependencies (GKE clusters, Vertex AI workspaces, GPU node pools, IAM bindings), a corrupted or out-of-sync state file causes Terraform to attempt to recreate production resources, causing outages. Best practices: (1) store state in a remote backend with locking (GCS bucket + google\_storage\_bucket with uniform bucket-level access, or Terraform Cloud); (2) use separate state files per environment (dev/staging/prod) — never share a state file across environments; (3) use terraform import to bring existing manually-created resources under Terraform management before modifying them; (4) enable state versioning on the GCS bucket — roll back to a previous state on plan divergence.

**Q69. How do you design Helm charts for AI inference workloads on GKE?**

**A:** Structure the Helm chart with AI-specific values: (1) values.yaml exposes model.name, model.version, model.gcsPath, gpu.type, gpu.count, replicaCount, and inference.maxTokens; (2) the Deployment template includes a resources.limits block with nvidia.com/gpu requests, nodeSelector for GPU node pool targeting, and a tolerations block for GPU node taints; (3) an init container runs gsutil -m cp gs://model-bucket/... to pre-load weights into the PVC; (4) a PodDisruptionBudget resource ensures at least 1 replica stays available during node upgrades; (5) expose a ServiceMonitor resource for Prometheus scraping of GPU and inference metrics. Store model version as a Helm label to enable clean rollback with helm rollback.

**Q70. How do you implement GitOps for an AI platform with ArgoCD?**

**A:** GitOps treats Git as the single source of truth for cluster state. For AI platforms: (1) organise the repository as apps/{env}/{service}/ — each AI service (embedding-api, inference-api, vector-db) has its own ArgoCD Application manifest pointing to a Helm chart; (2) environment promotion via pull requests — a PR from staging to prod branch triggers an ArgoCD sync to production; (3) model version updates are committed to values.yaml as a Git commit — ArgoCD detects the diff and rolls out the new model deployment; (4) ArgoCD's _automated sync with self-heal_ prevents configuration drift — if someone manually patches a Kubernetes resource, ArgoCD reverts it to match Git within 3 minutes; (5) use ArgoCD Image Updater to automatically commit new container image tags when CI pushes a new build.

MLOps & Model Lifecycle
-----------------------

**Q71. What is model drift and how do you detect and respond to it in production?**

**A:** Model drift is degradation in a deployed model's performance over time due to changes in the real-world data distribution. Two types: (1) _data drift (covariate shift)_ — the distribution of input features changes (e.g., users start asking different types of questions); detect with statistical tests — Population Stability Index (PSI), Kolmogorov-Smirnov test, or Jensen-Shannon divergence on feature distributions; (2) _concept drift_ — the relationship between inputs and correct outputs changes (e.g., a fraud pattern evolves); detect with label monitoring and periodic held-out evaluation sets. Response: if drift exceeds a threshold, trigger automated retraining on a fresh data window. Tools: Evidently AI, WhyLogs, or Vertex AI Model Monitoring.​

**Q72. Design a CI/CD pipeline for a machine learning model from training to production.**

**A:** Full ML CI/CD pipeline stages: (1) _Code CI_ — on PR, lint Python, run unit tests for feature engineering and training code, check data schema compatibility; (2) _Model training_ — triggered on merge to main; runs training pipeline on Vertex AI Pipelines or Kubeflow; outputs model artefact + evaluation metrics to MLflow/Vertex AI Experiments; (3) _Model evaluation gate_ — automated comparison against the current production model on a held-out test set; PR is blocked if new model degrades by > 2%; (4) _Container build_ — model served in a versioned Docker image pushed to Artifact Registry; (5) _Staging deploy_ — ArgoCD promotes to staging, automated integration tests run; (6) _Production promote_ — manual approval gate (or automated if staging tests pass) triggers Helm rollout with traffic shifting.​

**Q73. What is LoRA fine-tuning and when should you use it over full fine-tuning?**

**A:** LoRA (Low-Rank Adaptation) freezes the base model's weights and inserts small trainable low-rank matrices (rank 4–64) into the transformer's attention layers. Only these matrices (~0.1–1% of total parameters) are updated during training, dramatically reducing GPU memory and training time. Use LoRA when: (1) you want domain-specific tone or output format (not factual knowledge injection — use RAG for that); (2) you're working with a GPU budget that can't afford full fine-tuning (a 70B model full fine-tune requires 8× A100s; LoRA can run on a single A100 with QLoRA quantisation); (3) you need _multiple task-specific adapters_ on the same base model — each LoRA adapter can be swapped in and out without reloading the base model weights.

**Q74. What is RLHF and how does it improve LLM alignment?**

**A:** Reinforcement Learning from Human Feedback (RLHF) is a three-stage process: (1) _supervised fine-tuning (SFT)_ — fine-tune the base LLM on a curated dataset of high-quality human-written examples; (2) _reward model training_ — human labellers rank pairs of model outputs by quality; a separate reward model is trained to predict human preference scores; (3) _RL optimisation_ — the SFT model is further trained using PPO (Proximal Policy Optimisation) to maximise the reward model's score, subject to a KL-divergence penalty that prevents the model from deviating too far from the SFT baseline. RLHF is what makes raw LLMs into instruction-following assistants (GPT-4, Claude) — without it, they predict next tokens but don't follow user intent helpfully.​

**Q75. What is MLflow and how does it support the model lifecycle?**

**A:** MLflow is an open-source platform that tracks four stages of the ML lifecycle: (1) _Tracking_ — logs parameters, metrics, and artefacts for every training run, enabling experiment comparison; (2) _Models_ — standardises model packaging with a MLmodel spec, supporting flavours (sklearn, PyTorch, Hugging Face) for framework-agnostic serving; (3) _Registry_ — a versioned model store with lifecycle stages (Staging → Production → Archived) and approval workflows; (4) _Projects_ — reproducible training code packaging. For GCP shops, Vertex AI Experiments and Vertex AI Model Registry provide equivalent functionality natively integrated with BigQuery, Cloud Storage, and GKE — often the better choice when staying within the GCP ecosystem.

**Q76. What is the difference between online, offline, and near-real-time feature serving?**

**A:** Feature serving mode determines the freshness and latency trade-off: _offline_ features are precomputed and stored in a data warehouse (BigQuery, Snowflake) — high latency (minutes to hours), used for batch inference and training; _online_ features are stored in a low-latency key-value store (Redis, Bigtable) — sub-millisecond lookup, used for real-time inference; _near-real-time_ features are computed from streaming events (Kafka → Flink/Dataflow) and written to the online store with < 60 second lag — the best of both worlds for fraud detection or personalisation. A _feature store_ (Feast, Tecton, Vertex AI Feature Store) unifies all three serving modes with a single feature definition, eliminating training-serving skew.​

**Q77. How does Blue/Green deployment differ from Canary deployment for AI model releases?**

**A:** _Blue/Green_: two identical environments, 100% traffic on Blue. On new model release, deploy to Green. Switch all traffic from Blue to Green at once. Rollback = switch traffic back instantly. Zero-downtime, but no gradual quality validation — you won't catch quality regressions until all users are affected. _Canary_: gradually shift traffic (1% → 5% → 25% → 100%) to the new model version while monitoring quality, latency, and error rates. Catch problems before they affect the majority of users. For AI models — where _behavioural_ regressions are harder to detect than infrastructure failures — _canary is strongly preferred_. Use automated evaluation on the canary traffic slice to catch quality regressions before promoting to 100%.​

**Q78. Explain what a feature store is and why it matters for AI platforms.**

**A:** A feature store is a centralised repository that manages the creation, storage, and serving of ML features. Without it, every ML team builds its own feature pipelines, resulting in duplicated logic, training-serving skew (features computed differently in training vs inference), and no reuse across teams. A feature store solves this with: (1) _centralised feature definitions_ — computed once, reused by all teams; (2) _point-in-time correct retrieval_ — training datasets always reflect what was known at the time of each label, not future data; (3) _online/offline consistency_ — the same feature logic serves both training (offline) and inference (online) paths; (4) _feature discovery catalogue_ — engineers browse and reuse existing features. Vertex AI Feature Store, Feast (open-source), and Tecton (managed) are the main options.​

**Q79. What is model explainability and how do you implement it for an AI system in a regulated industry?**

**A:** Explainability answers "why did the model produce this output?" — critical for compliance (APRA CPS 230, EU AI Act) and trust. Approaches by model type: for _traditional ML_ (XGBoost, LightGBM), use SHAP (SHapley Additive exPlanations) to compute per-feature contribution scores — Vertex AI Explainable AI supports SHAP natively; for _deep learning_, use LIME (Local Interpretable Model-Agnostic Explanations) or attention weight visualisation; for _LLMs_, explainability is harder — use _citation grounding_ (show which retrieved document the answer came from), _confidence scores_, and _chain-of-thought traces_. Store explanations alongside predictions in an audit log — regulators increasingly require that consequential automated decisions (credit, insurance, hiring) be explainable on demand.

**Q80. How do you containerise a GPU-accelerated LLM inference service?**

**A:** Base image: nvidia/cuda:12.4.0-runtime-ubuntu22.04 — never use the devel image in production (3× larger). Install vLLM or TensorRT-LLM as the serving backend. Critical Dockerfile patterns: (1) copy model weights from GCS during build using a multi-stage build or load dynamically from a PVC at runtime (prefer PVC to avoid baking weights into the image); (2) set ENV CUDA\_VISIBLE\_DEVICES=0 or rely on Kubernetes GPU device plugin; (3) expose a /health endpoint that verifies GPU availability using nvidia-smi — Kubernetes liveness probe hits this; (4) configure ulimits for maximum open file descriptors (LLM inference opens many memory-mapped files); (5) set a non-root user for container security — USER 1001. Push to Artifact Registry with image digest pinning, never latest tags in production.

Prompt Engineering & LLM Internals
----------------------------------

**Q81. What is Chain-of-Thought (CoT) prompting and when does it significantly improve outputs?**

**A:** Chain-of-Thought prompting instructs the LLM to reason step-by-step before producing a final answer, by including phrases like "Let's think step by step" or showing reasoning examples in few-shot demos. CoT significantly improves performance on tasks requiring _multi-step reasoning_: maths problems, logical deduction, multi-hop fact retrieval, and code debugging. It is less effective for simple factual lookups or classification tasks — in those cases it adds latency and token cost without quality gain. For production use: _zero-shot CoT_ ("think step by step") is sufficient for most reasoning tasks; _few-shot CoT_ with curated examples is worth the prompt engineering investment for high-value, high-complexity pipelines like contract analysis or financial modelling.​

**Q82. What is the difference between zero-shot, one-shot, and few-shot prompting?**

**A:** _Zero-shot_: give the model a task instruction with no examples — relies entirely on the model's pre-trained knowledge ("Classify this review as positive or negative: ..."). _One-shot_: provide one example of the desired input-output format before the actual task. _Few-shot_: provide 3–10 carefully chosen examples. More examples generally improve performance on specialised tasks where the model lacks sufficient training data, but have diminishing returns beyond 5–10 examples and consume valuable context window tokens. For enterprise AI: few-shot is the fastest path to domain-specific output formatting; fine-tuning is the right solution if you need more than 10 examples to get consistent results — that threshold signals a fine-tuning use case.​

**Q83. How do you manage and version prompts in a production AI system?**

**A:** Treat prompts as code: store them in Git with semantic versioning. Each prompt has a unique ID, a version tag, and associated evaluation results. Use a _prompt management platform_ (LangSmith, PromptLayer, or Humanloop) that: (1) stores prompt versions with diffs; (2) associates each deployed prompt version with its quality benchmark scores; (3) enables A/B testing between prompt versions on live traffic; (4) supports feature-flag-driven rollout of new prompt versions. Rollback is a Git revert. Critical practice: _never hard-code prompts in application code_ — externalise them to a prompt registry so they can be updated without code deployment. This is especially important for regulated environments where prompt changes may require compliance review before rollout.

**Q84. What is temperature in LLMs and how should you configure it for different AI use cases?**

**A:** Temperature controls the randomness of the token sampling distribution. At temperature=0, the model always picks the highest-probability token (fully deterministic); at temperature=1.0, sampling follows the raw model distribution; at temperature > 1.0, outputs become increasingly random. Configuration guidelines:

Use CaseRecommended TemperatureFactual Q&A, RAG0.0 – 0.1Structured data extraction0.0Code generation0.1 – 0.2Summarisation0.3 – 0.5Creative writing, brainstorming0.7 – 1.0

Always pair low temperature with top\_p=1.0; pair higher temperature with top\_p=0.9 for quality control.

**Q85. How do you implement token budget management to control AI API costs at the application layer?**

**A:** Token cost = (input tokens + output tokens) × price per token. Control each component: (1) _input token control_ — enforce a max context window per request; implement prompt compression using LLMLingua or similar to remove redundant tokens from retrieved context; truncate conversation history to the last N turns beyond a threshold; (2) _output token control_ — always set max\_tokens on every API call — never leave it at the default maximum; (3) _per-tenant budgets_ — maintain a token counter per tenant in Redis; reject requests or downgrade to a cheaper model when the budget is near exhaustion; (4) _async batch pricing_ — for non-real-time workloads, use OpenAI Batch API (50% cost discount) or Vertex AI batch prediction.

**Q86. What is embedding model selection criteria for a production RAG system?**

**A:** Key selection criteria: (1) _dimensionality_ — higher dimensions (1536, 3072) capture more semantic nuance but cost more in storage and compute; 768-dim models often achieve 95% of the quality at 50% the cost; (2) _max sequence length_ — for long documents, you need models that support 8K+ tokens (e.g., text-embedding-3-large, bge-m3); (3) _domain specificity_ — a fine-tuned domain model (legal, medical, finance) outperforms a general model significantly on in-domain retrieval; (4) _multilingual support_ — for ANZ enterprises with APAC coverage, multilingual-e5-large or bge-m3 handles 100+ languages in one model; (5) _cost_ — self-hosted bge-large-en-v1.5 on a single CPU instance handles ~100 embeddings/second at near-zero marginal cost vs. $0.13/million tokens for text-embedding-3-large.

**Q87. What is LLM hallucination and what architectural patterns reduce it?**

**A:** Hallucination is when an LLM generates confident-sounding but factually incorrect content not grounded in its training data or provided context. Root causes: the model interpolates between learned patterns when it lacks knowledge, rather than saying "I don't know." Architectural mitigations: (1) _RAG with citation enforcement_ — instruct the model to only answer from retrieved context, and return source citations; (2) _confidence calibration_ — prompt the model to rate its own confidence; route low-confidence responses to a "I'm not sure" fallback or human escalation; (3) _output verification_ — a second LLM call verifies the first response against the retrieved context (using an "Is this claim supported by the provided documents?" check); (4) _constrained output domains_ — for structured tasks, use constrained decoding to limit outputs to known valid values.

**Q88. What is the difference between a system prompt, user prompt, and assistant message in LLM APIs?**

**A:** The message structure defines trust levels and behavioural scope: _system prompt_ is the highest-privilege message — set by the application developer, invisible to the end user in production, and used to define the model's persona, guardrails, output format, and scope constraints. It persists for the entire conversation. _User messages_ are end-user inputs — treated as lower-trust and potentially adversarial (prompt injection risk). _Assistant messages_ are the model's previous outputs included in the conversation history to maintain context. Architectural rule: _never inject raw user-supplied content directly into the system prompt_ — this is the primary vector for prompt injection. Always maintain strict boundary separation between system (developer-controlled) and user (end-user-controlled) message roles.

Data Architecture & Pipelines
-----------------------------

**Q89. How do you design an event-sourced audit trail for AI system decisions?**

**A:** Event sourcing stores every state change as an immutable event rather than updating current state. For AI audit trails: every inference request is stored as an event: {timestamp, userId, tenantId, promptHash, modelVersion, retrievedChunkIds, response, evaluationScore}. Events are written to an append-only store (Kafka → BigQuery, or Cloud Spanner for strong consistency). This provides: (1) _full reproducibility_ — replay any past decision with the exact context that produced it; (2) _regulatory compliance_ — auditors can inspect any AI decision, including what context the model was given; (3) _GDPR/Privacy Act right to erasure_ — implement a "crypto shredding" pattern where events referencing a user are encrypted with their personal DEK; deleting the DEK effectively erases the data without mutating the event log.

**Q90. How do you implement data lineage for an AI data pipeline?**

**A:** Data lineage tracks where data came from, how it was transformed, and where it went — essential for debugging model quality issues and satisfying regulatory audits. Implement at three layers: (1) _pipeline lineage_ — use Apache Atlas, OpenLineage (with Marquez backend), or dbt's native lineage to track dataset-level transformations across your ingestion, feature engineering, and training pipelines; (2) _feature lineage_ — the feature store records which features were computed from which raw data sources, with transformation code versioned in Git; (3) _model lineage_ — MLflow or Vertex AI Model Registry records which training dataset version, feature set version, and code commit produced each model artefact. When a model starts hallucinating, trace lineage back to identify if a training data source was corrupted or changed.

**Q91. What is the difference between batch inference and online inference architectures?**

**A:** _Batch inference_: process large volumes of records on a schedule (nightly, hourly) — run Spark or Vertex AI batch prediction jobs, write results to BigQuery or Cloud Storage. Optimised for throughput and cost (spot instances, no latency SLA). Used for: document pre-processing, bulk classification, scheduled report generation. _Online inference_: serve individual requests with low latency on-demand — deployed as a persistent service (Cloud Run, GKE + vLLM) with load balancing. Optimised for latency and availability. Higher cost per prediction due to always-on infrastructure. _Hybrid pattern_: batch-pre-compute embeddings and classifications for all known entities, store in a fast lookup store (Redis/Bigtable); online inference then only runs for genuinely novel inputs — dramatically reducing online inference volume and cost.

**Q92. How do you design a data ingestion pipeline for a multi-source RAG knowledge base?**

**A:** Multiple source types need different connectors: (1) _document sources_ (SharePoint, Google Drive, Confluence) — use webhook-triggered or scheduled connectors (Airbyte, custom Cloud Functions) to detect new/updated documents and push events to a raw-documents Pub/Sub topic; (2) _database sources_ — CDC (Change Data Capture) via Debezium streams row-level changes to the same topic; (3) _web sources_ — a scheduled scraper with scrapy or Firecrawl publishes crawled content; all sources converge on the same processing pipeline: extract text → chunk → embed → upsert to vector store with source metadata. Use a _document fingerprint_ (SHA-256 hash of content) to detect unchanged documents and skip reprocessing — essential for cost control at scale.

**Q93. What is PII detection and masking in an AI data pipeline and how is it implemented?**

**A:** PII (Personally Identifiable Information) must never reach an LLM in production environments handling sensitive data. Implementation: (1) _detection_ — use Google Cloud DLP API, AWS Comprehend, or Microsoft Presidio (open-source) to scan all input text for PII types (names, emails, phone numbers, tax file numbers, BSB/account numbers for ANZ); (2) _masking strategy_ — replace PII with pseudonymous tokens (\[NAME\_1\], \[ACCOUNT\_2\]) using a reversible vault mapping if you need to de-anonymise for human review; or use irreversible redaction for logs; (3) _pipeline placement_ — PII masking must run _before_ embeddings are generated and _before_ context is injected into LLM prompts; (4) _output scanning_ — post-generation, scan LLM outputs for PII to catch memorised training data leakage.

AI Security & Governance
------------------------

**Q94. What is responsible AI and how do you operationalise it in an engineering team?**

**A:** Responsible AI is the practice of building AI systems that are fair, accountable, transparent, and safe. Operationalising it means embedding it into the _engineering process_, not treating it as a post-hoc review: (1) _model cards_ — for every deployed model, publish documentation covering intended use, known limitations, fairness evaluation results, and out-of-scope uses; (2) _bias testing_ — run disaggregated evaluation across demographic slices before production deployment; (3) _red-teaming_ — a dedicated team probes the system for harmful outputs, jailbreaks, and discriminatory patterns; (4) _AI impact assessments_ — mandatory for high-risk use cases (hiring, lending, healthcare) before development begins; (5) _human-in-the-loop thresholds_ — define confidence thresholds below which AI decisions route to human review automatically.

**Q95. How do you perform threat modelling specifically for an LLM-powered application?**

**A:** Use the STRIDE framework adapted for LLM threats: (1) _Spoofing_ — can a user impersonate an admin via prompt injection? Mitigate: role enforcement at the system prompt + API authentication layer; (2) _Tampering_ — can retrieved documents be poisoned to alter model behaviour? Mitigate: document provenance validation and source trust scoring; (3) _Repudiation_ — can users deny having submitted a harmful prompt? Mitigate: immutable prompt logging with user identity binding; (4) _Information Disclosure_ — can the model be prompted to reveal other users' data or system prompt contents? Mitigate: prompt confidentiality instructions + output monitoring; (5) _Denial of Service_ — can an attacker submit max-token prompts to exhaust API budget? Mitigate: token budget limits per user session; (6) _Elevation of Privilege_ — can a user escalate from user role to admin via indirect prompt injection in retrieved documents?

Observability & Monitoring

**Q97. How do you implement distributed tracing for a multi-agent AI system?**

**A:** Each agent invocation in a multi-agent system must be traceable end-to-end. Use OpenTelemetry with a _trace context propagation_ pattern: (1) the orchestrator agent creates a root span with a unique trace\_id when a user request arrives; (2) every sub-agent call inherits the trace context as a child span — the trace\_id threads through all agent hops; (3) each span captures: agent name, tool calls made, token counts, model version, retrieved chunk IDs, latency; (4) traces are exported to a backend (Grafana Tempo, Datadog APM, Langfuse); (5) for async agent patterns (message-queue-based), propagate trace context via message headers. With this setup, a failing end-to-end request can be traced to the exact sub-agent, tool call, and retrieved context that caused the failure.

**Q98. What is embedding drift and how do you monitor for it?**

**A:** Embedding drift occurs when the statistical distribution of input embeddings shifts over time, indicating that users are asking different types of questions than the model was optimised for — a precursor to retrieval quality degradation. Detect it by: (1) computing a _reference distribution_ of embeddings from a historical baseline window (e.g., last 30 days); (2) sampling live query embeddings daily and computing _mean cosine distance_ between live and baseline clusters; (3) using dimensionality reduction (UMAP/PCA) to visualise cluster shifts on a monitoring dashboard; (4) alerting when Jensen-Shannon divergence between daily and baseline embedding distributions exceeds a threshold. Respond to confirmed embedding drift by: re-evaluating retrieval quality on a golden test set, and potentially re-embedding the knowledge base with an updated embedding model.​

**Q99. How do you implement cost allocation and FinOps reporting for an AI platform with multiple business units consuming shared infrastructure?**

**A:** Implement multi-layer cost attribution: (1) _infrastructure tagging_ — every GCP resource tagged with team, product, env, and cost-centre labels; BigQuery billing export enables per-label cost queries; (2) _AI API cost attribution_ — route all LLM API calls through a proxy (LiteLLM, a custom FastAPI middleware) that logs tenant\_id, token\_count\_input, token\_count\_output, and model to BigQuery; join with pricing tables for per-tenant cost reports; (3) _chargeback dashboards_ — Looker Studio or Grafana dashboard showing monthly AI spend per team, model, and use case; (4) _budget alerts_ — per-project GCP budget alerts at 80% and 100% of monthly allocation; automated model downgrade triggered at 95% to prevent overrun. Review monthly in an AI FinOps committee.

**Q100. How do you architect a system to detect and respond to AI model security incidents (e.g., a jailbreak being actively exploited in production)?**

**A:** Build a _real-time AI security monitoring pipeline_: (1) _detection_ — a streaming consumer reads all inference logs from Pub/Sub; a fast classifier (fine-tuned for jailbreak patterns, harmful content, PII exfiltration attempts) runs on every prompt-response pair in near-real-time (< 5 second lag); (2) _alerting_ — a spike in classifier flags triggers a PagerDuty/OpsGenie alert with the offending session IDs; (3) _immediate response_ — a circuit-breaker feature flag can disable the affected AI endpoint within 60 seconds, routing users to a fallback "maintenance mode" response; (4) _investigation_ — the full trace (prompt, retrieved context, response, user session) is preserved in an immutable audit log for forensic analysis; (5) _remediation_ — update guardrail rules, deploy updated system prompt, and run a regression test before re-enabling the endpoint. Document the incident in a blameless post-mortem with architectural remediation items.

**Q134. Design an AI-powered API documentation generator for a developer platform.**
