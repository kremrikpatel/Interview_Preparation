01 Executive Summary & Business Problems

MYOB maintains a central internal documentation platform spanning HR policies, project documentation, and technical documentation. Two critical business problems drive this initiative.

Problem 1 — HR Policy Access

Staff struggle to locate relevant HR policies across hundreds of documents. Queries like "what is the parental leave policy?" require manual searching across multiple repositories, categories, and document versions, leading to outdated information being used and increased HR team workload from repetitive questions.

Problem 2 — Project Prioritisation

With dozens of active and proposed projects, technical leadership lacks a data-driven mechanism to identify and rank the top 10 high-priority projects. Decisions are currently made through ad-hoc meetings without systematic evaluation of risk, cost, criticality, and cross-project dependencies.

Proposed Solution

A self-evaluating RAG pipeline with hybrid search (dense vector + sparse BM25) that ingests MYOB's central documentation, serves natural-language queries with cited answers, and includes an LLM-as-Judge evaluation layer for continuous quality assurance. Two specialised agents sit atop this pipeline: an HR Policy Assistant and a Project Prioritisation Engine.

02 Ingestion Pipeline

The ingestion pipeline transforms raw documents from MYOB's central documentation into searchable vector and keyword indices, handling multiple formats, category-aware metadata, and incremental updates.

Architecture — Document Ingestion Pipeline

Pipeline Components — Detailed Breakdown

Step 1: Document Loader

The document loader connects to MYOB's central documentation sources via API connectors (Confluence REST API, Jira API, SharePoint Graph API, Git repositories). Each connector normalises content into a unified intermediate representation containing raw text, structural markers (headings, tables, lists), and source metadata (URL, author, last-modified timestamp, document category). PDF and DOCX files are parsed using Apache Tika with OCR fallback for scanned documents. The loader maintains a content-hash registry in PostgreSQL to detect changes on subsequent runs, ensuring only modified documents are re-processed.

Step 2: Chunking Engine

Documents are split using a hybrid chunking strategy. The primary method is semantic chunking using sentence-boundary detection with a target chunk size of 512 tokens and 64-token overlap. For structured documents (HR policies with numbered sections, technical runbooks with step sequences), the chunker respects document structure boundaries — never splitting mid-section. Each chunk retains a pointer to its parent document, section hierarchy, and positional index. A secondary "summary chunk" is generated per document using an LLM call to create a 2-3 sentence abstract, enabling document-level retrieval alongside passage-level retrieval.

Step 3: Metadata Enricher

Every chunk is tagged with structured metadata: category (HR, Project, Technical), subcategory (e.g. Leave Policies, Security Runbooks), document_owner, last_updated, confidentiality_level, and document_version. For project documents, additional fields include project_name, priority_score, risk_level, estimated_cost, and criticality_rating. An LLM-based classifier assigns categories where manual tagging is absent, using a few-shot prompt trained on MYOB's taxonomy.

Step 4: Embedding Generator

Chunks are embedded using OpenAI's text-embedding-3-large (3072 dimensions) or an equivalent open-source model such as BGE-M3 for cost-sensitive workloads. Embeddings are generated in batches of 100 with retry logic and rate-limit handling. Each embedding is stored alongside its metadata in the vector index (Pinecone or Weaviate), while the same chunk text is indexed into Elasticsearch for BM25 keyword search. Full documents and their metadata persist in PostgreSQL with S3-backed blob storage for originals.

Incremental Sync: The scheduler runs every 6 hours (configurable) and on webhook triggers. It compares content hashes against stored values, re-chunks and re-embeds only changed documents, and marks deleted documents as inactive in the index rather than purging them (soft-delete with 30-day retention).

03 Augmented Generation Process

How the LLM generates responses by augmenting its reasoning with retrieved context, including prompt construction, context window management, and citation grounding.

Architecture — Augmented Generation Flow

Prompt Engineering Strategy

The prompt constructor assembles four layers into a single completion request. The system prompt establishes the assistant's persona (MYOB Internal Knowledge Assistant), sets behavioural guardrails (no speculation beyond retrieved context, mandatory citations, confidentiality enforcement), and defines the output schema. Retrieved chunks are injected with their metadata as numbered context blocks, enabling the LLM to produce inline citations like [Source 1: HR Leave Policy §3.2]. The user query is passed verbatim to preserve intent, while output format instructions enforce structured JSON containing the answer text, citation list, confidence score (0-1), and a boolean hallucination flag.

Context Window Management

With 128K context windows available in modern LLMs, the pipeline budgets token allocation: 2K for system prompt, up to 8K for retrieved context (approximately 5-8 chunks at 512 tokens each), the user query, and 2K reserved for output. When retrieval returns more chunks than the budget allows, a cross-encoder re-ranker (e.g. ms-marco-MiniLM-L-12) scores chunk-query relevance and selects the top-K within budget. This prevents context dilution where too many marginally relevant chunks reduce answer quality.

Citation Grounding

Every factual claim in the response must cite a specific chunk by its source document and section. The prompt instructs the LLM to output "I don't have enough information" when retrieved context doesn't cover the query, rather than fabricating an answer. Post-generation, a citation validator checks that every cited chunk ID exists in the retrieved set and that the cited text actually appears in that chunk.

04 Retrieval Augmented Generation — Full Process

The end-to-end RAG process combining hybrid search, re-ranking, augmented generation, and self-evaluation into a unified pipeline.

Architecture — Complete RAG Pipeline with Hybrid Search

Hybrid Search Strategy

The hybrid approach combines two complementary retrieval methods. Dense retrieval using vector similarity excels at semantic matching — understanding that "maternity leave" and "parental leave entitlements" are related even without keyword overlap. Sparse BM25 retrieval catches exact-match cases that vectors sometimes miss, particularly for MYOB-specific acronyms, policy numbers, and project codes. Results from both methods are merged using Reciprocal Rank Fusion (RRF), which balances contributions from each without requiring calibrated scores. The fused candidate set is then re-ranked by a cross-encoder model that jointly attends to the query and each candidate, producing a final relevance ordering.

Query Understanding Layer

Before retrieval, the query processor performs three operations. Intent classification determines whether the query is factual (policy lookup), analytical (project comparison), or navigational (find a specific document). Query expansion using Hypothetical Document Embedding (HyDE) generates a synthetic answer passage and embeds it, often retrieving more relevant chunks than the raw query embedding. Category detection uses a lightweight classifier to pre-filter the index by document category (HR, Project, Technical), reducing the search space and improving precision.

05 Experiment-to-Production Pathways

The lifecycle from initial prototyping through evaluation gates to production deployment, including model registry, A/B testing, and rollback mechanisms.

Architecture — Experiment to Production Pathway

Experiment Phase

Data scientists and AI engineers work in Jupyter notebooks using LangChain or LlamaIndex as the orchestration framework. Each experiment varies one dimension at a time: chunking strategy (fixed-size vs semantic vs document-structure-aware), embedding model (OpenAI ada-3 vs BGE-M3 vs Cohere embed-v3), retrieval method (dense-only vs hybrid), re-ranker inclusion, and prompt template. All experiments are tracked in MLflow with parameters, metrics (retrieval recall@K, answer faithfulness, latency), and artifacts (prompt templates, config files).

Evaluation Gate

Before any configuration advances to staging, it must pass an automated evaluation suite. This includes RAGAS metrics (faithfulness, answer relevance, context precision, context recall), LLM-as-Judge scoring on a curated test set of 200+ question-answer pairs across all three document categories, and a human review panel of 3 domain experts who score a random sample of 50 responses. The gate requires: faithfulness ≥ 0.85, answer relevance ≥ 0.80, context precision ≥ 0.75, and human approval ≥ 90%.

Staging and Production

Staging deploys the candidate pipeline alongside the current production version (shadow mode). A/B testing routes 10% of live traffic to the candidate, comparing response quality, latency, and user satisfaction. Production uses blue-green deployment with feature flags for instant rollback. An auto-rollback trigger fires if evaluation scores drop below thresholds for 30 consecutive minutes. Drift detection monitors embedding distribution shift and document corpus changes, triggering re-experimentation when significant drift is detected.

06 Production Scenario — End-to-End User Query

A detailed walkthrough of a production query from user input to evaluated response, showing every system interaction.

Architecture — Production Query Lifecycle

Production Query Walkthrough

A staff member types "What is the parental leave policy?" into the MYOB Knowledge Assistant. The API gateway authenticates the session (SSO via Okta), applies rate limiting (60 requests/minute per user), and sanitises the input (stripping injection attempts). A trace ID is generated for end-to-end observability in LangFuse.

The query is embedded into a 3072-dimensional vector using text-embedding-3-large. Simultaneously, the query processor classifies the intent as "factual/policy-lookup" and the category as "HR", which narrows the retrieval scope. An optional HyDE step generates a hypothetical answer passage and embeds it for improved semantic matching.

Hybrid retrieval fires two parallel searches: vector similarity search returns the top-20 nearest chunks from the HR partition of the vector index, while BM25 keyword search returns the top-20 from the Elasticsearch HR index. Reciprocal Rank Fusion merges and deduplicates these into a single ranked list. The cross-encoder re-ranker then scores each candidate against the original query, selecting the top-5 chunks that exceed the 0.75 relevance threshold.

The prompt constructor assembles the system prompt, the 5 retrieved chunks (with metadata), the user query, and output format instructions into a single request to Claude Sonnet. The response is streamed back to the user with inline citations referencing specific policy documents and sections. Asynchronously, the LLM-as-Judge evaluates the response for faithfulness (does it contradict the retrieved context?), relevance (does it address the user's question?), and safety (no harmful content). Scores below threshold trigger a human review flag. The entire interaction is logged to LangFuse with latency breakdowns, token counts, and eval scores.

07 Evaluation & Monitoring

The self-evaluating architecture that ensures continuous quality through LLM-as-Judge, automated metrics, and human-in-the-loop feedback.

Architecture — Evaluation & Monitoring System

LLM-as-Judge Implementation

The self-evaluation layer runs asynchronously after each response is generated, using a separate LLM call (typically a smaller, faster model like Claude Haiku) with a structured evaluation prompt. The judge receives the original query, the retrieved chunks, and the generated response, then scores each dimension on a 1-5 scale with written rationale. Scores are normalised to 0-1 and logged to LangFuse. Responses scoring below 0.7 on any dimension are routed to a human review queue in Argilla, where domain experts can correct the response and add it to the golden test set, creating a continuous improvement flywheel.

Monitoring Strategy

Three monitoring tiers operate simultaneously. Real-time monitoring (LangFuse traces) captures every query with latency breakdowns per pipeline stage, token counts, retrieval scores, and eval results. Infrastructure monitoring (Datadog) tracks API latency percentiles, error rates, throughput, memory usage, and vector index health. Business monitoring (custom dashboards) analyses query patterns by category, identifies coverage gaps (frequent queries with low retrieval scores), and tracks user satisfaction trends. PagerDuty alerts fire when any metric crosses its threshold for 5+ consecutive minutes, routing to the on-call AI engineer.

Drift Detection: A weekly job compares the embedding distribution of new documents against the existing index. If the centroid shift exceeds a configurable threshold (default: 0.15 cosine distance), it triggers a re-indexing alert and flags the document category for review. This catches cases where document language or structure has changed significantly, degrading retrieval quality.

08 Business Solution: HR Policy Access

Addressing Problem 1 — how staff can easily access HR policies through natural language queries with authoritative, cited answers.

Architecture — HR Policy Assistant

How It Solves the Problem

Staff currently waste 15-20 minutes searching across multiple systems for HR policy information, and frequently reference outdated versions. The HR Policy Assistant provides instant, accurate answers via channels staff already use (Slack, Teams, web portal). Every response cites the specific policy document, section number, and version, so staff can verify the source. The system restricts its answers strictly to content found in the HR knowledge base — when a question falls outside its scope, it escalates directly to the HR team with the query context, so the employee still gets help without the assistant fabricating an answer.

Key Design Decisions

The HR agent uses a specialised system prompt that emphasises precision over creativity (temperature 0.0), mandatory citation of policy sections, and explicit version-date disclosure. A version validator cross-references the cited document version against the latest version in the document store, flagging if the retrieved chunk comes from a superseded document. The intent router distinguishes between different HR domains (leave, benefits, conduct, WHS) and applies category-specific metadata filters to improve retrieval precision. For sensitive topics (disciplinary procedures, termination, grievances), the system immediately provides a brief overview and recommends speaking with HR directly, reducing liability risk.

Multi-Channel Deployment

Slack bot integration (slash command + conversational), Microsoft Teams app, and a dedicated web portal. All channels share the same backend RAG pipeline, ensuring consistent answers regardless of access point.

Confidentiality Controls

Role-based access filtering ensures managers see management-level policies (e.g. performance management procedures) while all-staff queries return general policies. Implemented via metadata filters on confidentiality_level field mapped to the user's role from Okta.

Freshness Guarantee

HR documents are re-ingested within 1 hour of any update (webhook from Confluence/SharePoint). Stale responses (citing documents older than 90 days) trigger a warning badge and automatically notify the HR content owner to review.

Analytics for HR Team

Dashboard showing top-queried topics, unanswered queries (coverage gaps), query volume trends, and user satisfaction. HR can identify which policies need clearer documentation based on confusion patterns.

09 Business Solution: Project Prioritisation Engine

Addressing Problem 2 — using RAG and LLM to summarise and rank the top 10 high-priority projects for technical leadership, evaluated across risk, cost, criticality with detailed trade-off analysis.

Architecture — Project Prioritisation Engine

Multi-Criteria Scoring Methodology

Each project is scored across four weighted dimensions extracted from its documentation. Risk (30% weight) evaluates technical complexity, dependency fragility, team capacity constraints, and historical delivery performance. Cost (20% weight) encompasses budget allocation, resource-hours required, infrastructure costs, and projected ROI. Criticality (35% weight, the highest weight) assesses business impact if the project is delayed, regulatory compliance deadlines, revenue implications, and customer-facing severity. Strategic alignment (15% weight) measures fit with MYOB's current OKRs, executive priorities, and market timing windows.

The scoring engine first uses RAG retrieval to pull relevant sections from each project's documentation (scope documents, risk registers, budget estimates, status reports). An LLM then extracts structured scores from this context, grounding each score in specific evidence from the documents. The composite score is calculated as: Composite = 0.30×Risk + 0.20×Cost + 0.35×Criticality + 0.15×Strategic.

Top 10 Selection and Ranking

Projects are sorted by composite score, and the top 10 are selected for detailed analysis. For each selected project, the LLM generates a structured summary containing the project name and owner, a 3-sentence executive summary, the individual dimension scores with evidence citations, a dependency map (which other projects it blocks or is blocked by), and team capacity assessment. The LLM also identifies potential conflicts between the top 10 (e.g., two projects requiring the same team at the same time) and suggests sequencing adjustments.

Trade-off Analysis Framework

For each of the top 10 projects, the engine produces a detailed trade-off analysis explaining both the case for picking up the project and the consequences of deferral.

Dimension Pick-Up Analysis Deferral Analysis
Risk What risks does starting this project mitigate? What new risks does it introduce (technical debt, team burnout, parallel work contention)? How does the risk profile change over time if deferred? Does the risk compound (e.g., security vulnerabilities), or remain stable?
Cost Total cost of ownership including team allocation, infrastructure, and opportunity cost of the team not working on alternatives. Expected ROI timeline. Does deferral reduce or increase cost? (Some projects become more expensive to deliver later due to accumulated technical debt or vendor price changes.)
Criticality What business outcomes depend on this project? Are there compliance deadlines, contractual obligations, or revenue milestones tied to delivery? What is the business impact of delay? Is there a hard deadline (regulatory) or a soft deadline (market window)? What workarounds exist?
Dependencies How many downstream projects does this unblock? What is the cascading impact of completing this project on the broader portfolio? Which teams or projects are blocked while this is deferred? Can they proceed with partial workarounds?
Example Output — Project #1: API Gateway Migration

Why pick up: The current API gateway has a known CVE with a CVSS score of 9.1, and the vendor's security patch requires the newer version. Compliance audit in Q3 2026 requires remediation evidence. Six downstream services depend on gateway features only available in the new version, currently blocked. Estimated 3-month delivery with 4 engineers.

Why not pick up (trade-offs): Migrating now pulls 4 senior engineers from the Data Platform v2 project for 3 months, delaying that project's revenue-generating features. The security risk can be partially mitigated with a WAF rule (temporary, 60% coverage). If the team is split, both projects risk quality degradation. Recommendation: Prioritise due to compliance deadline. Mitigate Data Platform delay by hiring 2 contract engineers for the overlap period.

10 Trade-off Analysis & Decision Framework

Architectural decisions and their trade-offs across the entire RAG pipeline.

Decision Option Chosen Alternative Trade-off Rationale
Search Strategy Hybrid (Dense + BM25) Dense-only or BM25-only Higher latency (+50ms) and infrastructure cost (two indices), but 15-20% improvement in retrieval recall, especially for MYOB-specific terminology that vectors alone miss.
Embedding Model text-embedding-3-large BGE-M3 (self-hosted) API dependency and per-token cost, but consistently higher retrieval quality (3% better recall@10 in benchmarks) and zero infrastructure management. Self-hosted option available for cost reduction at scale.
Chunking Semantic (512 tok, 64 overlap) Fixed-size (256 or 1024) More complex implementation and slightly higher compute, but respects document structure boundaries and reduces mid-thought splits by 40%, improving answer coherence.
Re-ranker Cross-encoder (MiniLM-L-12) No re-ranker / LLM re-rank Adds 100-150ms latency per query, but improves precision@5 by 25%. LLM re-ranking would be more accurate but 10x more expensive and 5x slower.
Evaluation LLM-as-Judge (async) Rule-based or human-only Adds ~$0.02/query in LLM cost, but scales to 100% of queries (vs. human review at 1-5%). Async execution means zero impact on user-facing latency.
Vector Database Managed (Pinecone/Weaviate Cloud) Self-hosted (Milvus, Qdrant) Higher per-query cost but zero operational burden. For MYOB's document volume (est. 50K-100K chunks), managed tier is cost-effective. Migrate to self-hosted if volume exceeds 1M chunks.
LLM Provider Claude Sonnet (primary) GPT-4o, Gemini, open-source Best balance of quality, instruction-following (critical for citation compliance), and cost. GPT-4o as fallback provider for resilience. Open-source (Llama 3) evaluated for cost-sensitive batch operations (project scoring).
Deployment Blue-green with feature flags Canary, rolling update Slightly more infrastructure (two full environments), but enables instant rollback within seconds vs. minutes for canary. Critical for an internal tool where bad answers erode trust quickly.

Evaluation Costs
