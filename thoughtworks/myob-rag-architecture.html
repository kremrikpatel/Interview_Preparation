01. Executive Summary & Business Problems

MYOB maintains a large corpus of internal documentation spanning HR policies, project documentation, and technical knowledge bases. Two critical business problems have been identified that this RAG architecture directly addresses.

Problem 1

HR Policy Access Gap

Staff across MYOB struggle to locate and interpret the correct HR policies. Policies are scattered across multiple repositories, versioned inconsistently, and written in dense legal language. Employees often contact HR teams directly for simple questions, creating bottlenecks and delays. The average resolution time for policy-related queries is 2–4 business days, and incorrect self-interpretation of policies leads to compliance risks.

Desired outcome: A conversational interface where any MYOB employee can ask HR policy questions in natural language and receive accurate, cited answers within seconds.

Problem 2

Project Prioritisation Complexity

MYOB's technical teams manage hundreds of active project documents. Leadership requires data-driven prioritisation of the top 10 high-priority projects based on risk, cost, criticality, and strategic alignment. Today, this requires manual review by multiple stakeholders, taking weeks and producing subjective, inconsistent rankings. Trade-off analysis between projects is rarely documented.

Desired outcome: An AI-driven system that ingests all project documentation, evaluates each project against defined criteria (risk, cost, criticality), and produces a ranked top-10 list with detailed justifications, trade-off analysis, and actionable summaries for technical leadership.

Solution Approach Both problems are addressed by a unified Self-Evaluating RAG Pipeline that combines hybrid search (dense vector + sparse keyword), LLM-powered generation with chain-of-thought reasoning, and an LLM-as-Judge evaluation layer that continuously monitors answer quality, faithfulness, and relevance.

02. System Architecture Overview

The high-level architecture connects four major subsystems: ingestion, retrieval, generation, and evaluation. Each subsystem is independently scalable and observable.

Figure 1 — High-Level System Architecture

The architecture follows a modular, event-driven design. Each component can be developed, tested, and scaled independently. The evaluation layer operates as a sidecar to generation, scoring every response before it reaches the user and feeding quality signals back into the system for continuous improvement.

03. Ingestion Pipeline

The ingestion pipeline is responsible for transforming raw documents from multiple sources into searchable, semantically-rich vector embeddings and sparse indices. It operates both as a batch process (initial load) and as an incremental pipeline (change detection).

Figure 2 — Ingestion Pipeline Architecture

Pipeline Stages in Detail

Stage 1 — Source Connectors

Connectors are adapter modules for each data source. For MYOB, the primary connectors target Confluence (HR policies and project docs), SharePoint (uploaded policy PDFs), JIRA (project metadata, status, risk flags), Git repositories (technical documentation in Markdown), and the HR management system (policy versioning metadata). Each connector implements a standard interface: list(), fetch(doc_id), and get_changes_since(timestamp). The change-detection method enables incremental ingestion — only new or modified documents trigger re-processing.

Stage 2 — Pre-Processing

Raw documents arrive in heterogeneous formats (PDF, DOCX, HTML, Markdown, JIRA JSON). The pre-processor normalises all content into a unified intermediate representation. For scanned PDFs (common in older HR policy archives), OCR via Tesseract or Azure Document Intelligence extracts text. Table structures are preserved as structured Markdown to maintain the relationships in policy matrices or project RACI charts. A hash-based Change Data Capture (CDC) mechanism compares document hashes against previously-ingested versions, skipping unchanged documents and flagging updated ones for re-chunking.

Stage 3 — Chunking Strategy

Chunking is critical for retrieval quality. The pipeline uses a multi-strategy approach tailored to document type:

Document TypeChunking StrategyChunk SizeOverlapRationale
HR PoliciesSemantic (section-aware)512 tokens64 tokensPolicies have clear section boundaries; smaller chunks improve precision for specific clause retrieval
Project DocsParent-child hierarchicalParent: 2048, Child: 512128 tokensProject context requires broad understanding; child chunks enable precise retrieval while parent provides context
Technical DocsSliding window + code-aware1024 tokens128 tokensCode blocks and technical explanations need contiguous context
JIRA TicketsPer-ticket (whole document)VariableN/ATickets are typically short; keeping them whole preserves issue-context relationships

Every chunk is enriched with metadata: source document ID, category (HR/Project/Technical), sub-category, version, last-modified date, author, access-control tags, and an LLM-generated summary of the chunk's content (used for hypothetical document embeddings in the retrieval stage).

Stage 4 — Embedding & Indexing

Each chunk is embedded using a dual-representation strategy. Dense embeddings are generated via text-embedding-3-large (3072 dimensions) for semantic similarity matching. Sparse vectors are generated using SPLADE or BM25 for lexical keyword matching — essential for exact policy names, acronym matches, and specific project codes that dense embeddings may miss. Both representations are upserted in batch to their respective stores, with chunk-to-source lineage recorded for citation generation.

Key Design Decision — Hybrid Embedding Pure dense retrieval fails on MYOB-specific acronyms (e.g., "PSL" for Personal/Sick Leave) and exact policy clause references ("Section 4.2.1"). Pure sparse retrieval misses semantic intent ("Am I allowed to work from another country?"). The hybrid approach captures both, with Reciprocal Rank Fusion (RRF) merging results at query time.

04. Retrieval Augmented Generation Process

The RAG process orchestrates the interplay between retrieval and generation. This section details the full pipeline from raw user query to ranked, retrieved context chunks ready for the LLM.

Figure 3 — RAG Retrieval Process (Hybrid Search)

Query Processing Pipeline

Intent Classification

An LLM-based classifier (lightweight fine-tuned model or few-shot prompt) categorises the incoming query into one of several intent types: HR_POLICY_LOOKUP, PROJECT_QUERY, PROJECT_PRIORITISATION, TECHNICAL_HOWTO, GENERAL_SEARCH. This classification drives downstream behaviour — HR queries apply policy-version metadata filters; project prioritisation queries trigger the multi-document analysis agent rather than simple retrieval.

Query Rewriting & Expansion

The original user query is rewritten for improved retrieval. For example, "can I WFH on Fridays?" is expanded to "MYOB work from home policy flexible working arrangements Friday remote work". A Multi-Query Generator produces 3–5 semantically diverse reformulations to increase recall. Hypothetical Document Embeddings (HyDE) generate a hypothetical "ideal answer" passage, which is then embedded — this shifts the query embedding closer to the answer space rather than the question space, significantly improving retrieval for complex or abstract queries.

Hybrid Search Execution

Two parallel search paths execute simultaneously. The dense path performs approximate nearest-neighbour (ANN) search against the vector database, returning the top-20 semantically similar chunks. The sparse path runs a BM25/SPLADE query against Elasticsearch, returning the top-20 lexically matched chunks. Results are merged using Reciprocal Rank Fusion (RRF) with a weighting of α=0.7 for dense and 0.3 for sparse — tuned empirically via offline evaluation datasets. The merged list is then passed through a cross-encoder reranker (e.g., ms-marco-MiniLM or Cohere Rerank) that performs pairwise relevance scoring between the query and each candidate chunk. The final top-5 chunks (configurable) are assembled into the LLM context window.

Why Hybrid over Pure Dense? In MYOB's domain, queries like "PSL entitlement for part-time staff under Section 12.3" require both semantic understanding ("PSL" = Personal/Sick Leave) and exact lexical matching ("Section 12.3"). Hybrid search achieves 18–23% higher NDCG@5 over pure dense retrieval in internal benchmarks.

05. Augmented Generation Process

Once relevant chunks are retrieved, the generation process constructs a grounded, cited response. This stage is where the LLM transforms raw retrieved context into a coherent, accurate answer.

Figure 4 — Augmented Generation Architecture

Prompt Engineering Strategy

The prompt is assembled programmatically from four components. The system prompt defines the assistant's role ("You are an internal MYOB knowledge assistant"), behavioural constraints (never fabricate policy details, always cite sources, flag low-confidence answers), and output formatting rules. Retrieved chunks are injected as numbered context blocks with their metadata, allowing the model to attribute claims. The chain-of-thought instruction directs the LLM to reason step-by-step before producing a final answer — particularly important for the project prioritisation use case, where multi-criteria evaluation requires explicit reasoning.

Post-Processing Pipeline

The raw LLM output passes through five post-processing stages before reaching the user. The Citation Linker validates that all cited sources actually exist in the retrieved context (rejecting hallucinated citations). The Hallucination Check runs a lightweight entailment model to verify that each claim in the response is supported by at least one retrieved chunk. The PII Scrubber detects and redacts any personally identifiable information that may have leaked through retrieved content. The Format Validator ensures the response follows the expected structure (structured JSON for the project prioritisation API; natural language with inline citations for HR queries). The Confidence Scorer aggregates retrieval scores, entailment confidence, and citation coverage into a single response confidence metric (0–1), which is surfaced to the user and logged for monitoring.

07. Experiment-to-Production Pathway

Moving from research notebooks to reliable production systems requires structured gating, versioning, and rollback capabilities. This pathway ensures that every change to the RAG pipeline — from embedding model swaps to prompt revisions — is evaluated before reaching users.

Figure 6 — Experiment-to-Production Pipeline

Version Management

Every component in the pipeline is independently versioned: embedding models (embed-v3.1), chunking configurations (chunk-hr-v2), prompt templates (prompt-hr-policy-v4), and LLM model versions. A deployment manifest captures the exact combination of component versions for each release, enabling precise rollback to any prior state. All experiments are tracked in MLflow with hyperparameters, metrics, and artifact links.

08. Evaluation & Monitoring (LLM-as-Judge)

The self-evaluating nature of this RAG pipeline is its defining characteristic. An independent LLM (the "Judge") evaluates every production response asynchronously, providing continuous quality signals without impacting user-facing latency.

Figure 7 — LLM-as-Judge Evaluation Architecture

Evaluation Metrics in Detail

MetricDefinitionHow MeasuredTarget
FaithfulnessEvery claim in the response is supported by the retrieved contextJudge LLM decomposes response into claims, checks each against context chunks≥ 0.90
Answer RelevanceThe response directly addresses the user's questionJudge generates N hypothetical questions from the answer, measures cosine similarity to original query≥ 0.85
Context PrecisionRetrieved chunks are relevant (no noise in context)Judge scores each retrieved chunk for relevance to the query≥ 0.80
Context RecallAll information needed to answer was retrievedJudge checks if the ground-truth answer is recoverable from retrieved chunks alone≥ 0.80
Hallucination RatePercentage of claims not grounded in retrieved contextInverse of faithfulness, computed per-claim< 5%
Safety / ToxicityResponse is free from toxic, biased, or inappropriate contentDedicated classifier + Judge review100% pass

Human-in-the-Loop Feedback

Every response includes a lightweight feedback mechanism (thumbs up/down + optional comment). Low LLM-as-Judge scores (below 0.75) automatically queue the response for human review. Human-corrected responses are added to the golden evaluation dataset, continuously improving the benchmark. Over time, the human review panel provides corrections that train the Judge itself, closing the self-improvement loop. User feedback and Judge scores are correlated to calibrate the Judge — if the Judge consistently rates a response type highly but users rate it poorly, the Judge prompts are revised.

Critical Design Decision — Judge Independence The Judge LLM must be a different model from the Generator LLM (or at minimum, a different prompt configuration). Using the same model to evaluate its own output creates blind-spot bias. In production, if the Generator is Claude Sonnet, the Judge should be GPT-4o (or vice versa) to ensure truly independent evaluation.

09. Business Application A — HR Policy Access

This section details how the RAG pipeline is configured specifically to solve MYOB's HR policy access problem, enabling staff to get instant, accurate, cited answers to any HR-related question.

Figure 8 — HR Policy RAG Application Flow

HR-Specific Configuration

Policy Version Management

HR policies are frequently updated. The system maintains version-aware indexing — each policy chunk carries a policy_version and effective_date metadata tag. At query time, only chunks from the current effective policy version are retrieved by default. If a user asks about historical policies ("What was the parental leave policy before January 2025?"), the version filter is relaxed to include the specified period. A nightly reconciliation job compares the latest policies in the HR system against the indexed versions, triggering re-ingestion for any updates.

Employee Context Enrichment

Where the user is authenticated (via SSO), the system can optionally enrich queries with employee context: employment type (full-time, part-time, casual), tenure, location (state/territory for jurisdiction-specific policies), and role level. This allows the LLM to tailor answers — for example, "Your parental leave entitlement as a part-time employee with 18 months tenure under the Victorian employment framework is..." This feature requires explicit user consent and is governed by MYOB's internal data privacy policies.

Guardrails for HR Content

The HR application includes additional guardrails beyond the standard pipeline. The LLM is instructed to never provide definitive legal advice, always include a disclaimer directing users to HR for complex or sensitive matters, never speculate on policy interpretation beyond what the text explicitly states, and always cite the specific policy section. If the confidence score falls below 0.7 or the query touches sensitive topics (disciplinary procedures, termination, discrimination complaints), the system automatically appends an escalation prompt: "For this type of question, we recommend contacting the HR team directly at hr@myob.com or via the internal HR portal."

Expected Business Impact Target: reduce HR team's policy-inquiry workload by 60–70%. First-response time drops from 2–4 business days to under 5 seconds. Employee satisfaction with HR information access projected to increase from 45% to 85% (measured via quarterly survey). Compliance risk reduced through consistent, version-controlled answers.

10. Business Application B — Project Prioritisation Engine

This application uses multi-document RAG with an agentic workflow to analyse all project documentation and produce a ranked top-10 project list with detailed justifications based on risk, cost, criticality, and strategic alignment.

Figure 9 — Project Prioritisation Agent Architecture

Scoring Framework

Risk Score (Weight: 30%)

The LLM agent analyses each project's risk register, technical specifications, and post-mortem data to compute a risk score. Factors include: technical complexity (novel technology, integration points), resource dependency (key-person risk, scarce skills), timeline pressure (deadline proximity, buffer remaining), external dependencies (vendor lock-in, regulatory changes), and historical risk materialisation (similar projects' outcomes). Each factor is scored 1–10 by the LLM with explicit reasoning, then weighted into a composite risk score.

Cost Score (Weight: 25%)

Cost analysis considers: total budget vs. spend-to-date (burn rate), projected cost-to-complete, ROI timeline, opportunity cost of delay, and cost of inaction. The LLM extracts financial data from budget documents and JIRA time-tracking, calculating a normalised cost-efficiency score. Projects with high ROI and low remaining cost score higher.

Criticality Score (Weight: 30%)

Criticality measures business impact: number of downstream dependencies, revenue impact if delayed, regulatory or compliance deadlines, customer-facing vs. internal, and strategic alignment with MYOB's annual objectives. The dependency map is particularly important — a project that blocks five others scores higher than one with no dependents, even if the latter has higher individual risk.

Strategic Alignment (Weight: 15%)

The LLM evaluates each project against MYOB's stated strategic priorities (provided as a reference document in the system prompt). Projects directly supporting top-3 strategic themes receive a multiplier. This prevents purely tactical projects from dominating the list.

Trade-off Analysis for Each Top-10 Project

For each project in the top 10, the agent produces a detailed trade-off analysis covering: what resources are consumed that could serve other projects, which lower-ranked projects are delayed as a consequence, what risks are accepted by prioritising this project, alternative approaches considered and why they were rejected, and specific action items for the technical team to begin work. This trade-off analysis is the key differentiator — it transforms the ranking from a simple ordered list into a decision-support tool that leadership can use to understand the second-order effects of their prioritisation decisions.

Important Caveat The project prioritisation engine provides data-driven recommendations, not decisions. All rankings and trade-offs should be reviewed by the technical leadership team and portfolio management board. The LLM's reasoning is fully transparent (chain-of-thought is included in the output) so decision-makers can audit the logic and adjust weights or override individual scores based on context the system may not have access to.

Composite Scoring Formula (Project Prioritisation)

Formula Composite = (0.30 × Risk) + (0.25 × Cost_Efficiency) + (0.30 × Criticality) + (0.15 × Strategic_Alignment)

Where each factor is normalised to 0–10. The weights are configurable by leadership and stored as a versioned configuration. Historical weight adjustments are tracked for audit.

Golden Evaluation Dataset Structure

The golden dataset contains 200+ curated question-answer pairs covering both HR policies and project documentation. Each entry includes: the user question, the ideal answer (human-authored), the expected source documents and sections, quality labels (faithfulness, relevance, completeness on 1–5 scale), and edge-case tags (multi-hop reasoning, numerical extraction, policy conflict resolution). This dataset is versioned alongside the pipeline and grows continuously from human review of flagged responses.

Security & Compliance Considerations

All data remains within MYOB's cloud tenancy — no document content leaves the organisation's security boundary. The LLM API calls transmit only chunked context (not full documents) and are governed by data processing agreements with the LLM provider. PII is scrubbed from logs and evaluation traces. Access control inherits from MYOB's existing RBAC: users can only retrieve documents they have permission to view in the source system. Embedding vectors are not invertible to original text, but are treated as sensitive data regardless.
